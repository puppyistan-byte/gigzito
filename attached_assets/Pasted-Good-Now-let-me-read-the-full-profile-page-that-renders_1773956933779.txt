Viewing a GeeZee Card — Mobile App Implementation
The card view screen requires 3 parallel API calls on load, then individual calls for each user action. Everything below is exactly what the web app uses.

Step 1: Load the Screen (3 calls, fire in parallel)

Call 3 — Follow status + follower count
GET /api/geezee-follows/status/{userId}
Auth: Required (JWT)
Response:

{
  "following": true,
  "followerCount": 42
}
Only make this call if the viewer is logged in. If not logged in, skip it — show a "Sign in to follow" prompt and followerCount will not be shown (or you can still call the endpoint anonymously, it will return 401, and you just omit the follower count from the UI).

Step 2: User Actions
Tap to Engage
Fire this whenever the viewer taps the card or an engage button. No body needed.

POST /api/gigness-cards/{card.id}/engage
Auth: Required (JWT)
Body: {}
Response: { "success": true }
Optimistically increment engagementCount in the UI, no need to reload the card.

Send Private Message
Only show this if card.allowMessaging === true and the viewer is not the card owner.

POST /api/gigness-cards/{card.id}/message
Auth: Required (JWT)
Body:
{
  "messageText": "Hey, I'd love to collaborate!",   // max 500 chars
  "emojiReaction": "🔥"                              // optional, max 8 chars
}
At least one of messageText or emojiReaction must be present.

Success: 201 { "success": true, "messageId": 42 }

Error (GZ-Bot flagged the text):

{ "message": "Hey! GeeZee stays PG. Clean up your pitch to engage with this card." }
Show this as an inline field error — let the user edit and retry, do not dismiss the form.

Follow / Unfollow
POST   /api/geezee-follows/{userId}    → Follow
DELETE /api/geezee-follows/{userId}    → Unfollow
Auth: Required
Body: {}
Response: { "ok": true }
After success, re-fetch GET /api/geezee-follows/status/{userId} to update the button and follower count.

Comments on a Geemotion (expandable thread under each post)
Load comments (lazy — only when user taps "Comments"):

GET /api/zee-motions/{motionId}/comments
Auth: Not required
Response:
[
  {
    "id": 5,
    "motionId": 34,
    "authorUserId": 9,
    "authorName": "alexrivera",
    "commentText": "Incredible work, congrats!",
    "createdAt": "2026-03-18T15:00:00.000Z"
  }
]
Post a comment:

POST /api/zee-motions/{motionId}/comments
Auth: Not required (name is provided manually if not logged in)
Body:
{
  "commentText": "Love this energy!",
  "authorName": "alexrivera"
}
If the user is logged in, resolve authorName automatically from their profile (displayName → username → email prefix). Do not ask them to type their name.

Response: 201 + comment object

QR Code / Deep Link
The card's QR UUID lives at card.qrUuid. On mobile, generate a QR code from:

https://gigzito.com/qr/{card.qrUuid}
Or pass the UUID to:

GET /api/gigness-cards/qr/{qrUuid}
Auth: Public
This is the entry point when someone scans a physical QR card — use it to load the card and redirect to the card view screen.

Recommended Load Sequence
1. Fire calls 1, 2, 3 in parallel on screen mount
2. Render skeleton placeholders while loading
3. Once card loads → render identity block, gallery, action buttons
4. Once motions load → render Geemotions feed below
5. Follow status → toggle Follow button state
6. Comments → lazy load per Geemotion only when tapped
