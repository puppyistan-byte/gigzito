There's no forgot-password route yet — the security module only handles change-password (while logged in) and MFA. Here's the full auth reference for the mobile developer:

Gigzito Auth & Security — Mobile Developer Reference
Base URL: https://www.gigzito.com
Email provider: Brevo SMTP (noreply@gigzito.com)
All requests: Content-Type: application/json
Mobile auth: returns a JWT Bearer token — store it securely and attach as Authorization: Bearer <token> on every subsequent request

REGISTRATION
POST /api/auth/register
Body:

{ "email": "user@example.com", "password": "minimum8chars" }
What happens: Creates account → Brevo sends a verification email with a link
Responses:

201 → { "user": { "id", "email", "role" }, "profile": null }
400 → { "message": "Email and password required" }
409 → { "message": "Email already registered" }
EMAIL VERIFICATION
GET /api/auth/verify-email?token=<token_from_email_link>
What happens: Marks account as verified — must be done before login is allowed
Rate limited: yes
Responses:

200 → { "message": "Email verified successfully" }
400 → { "message": "Invalid or expired verification token" }
POST /api/auth/resend-verification
Body:

{ "email": "user@example.com" }
What happens: Brevo resends the verification email
Rate limited: yes
Responses:

200 → { "message": "Verification email sent" }
LOGIN (2-STEP — WEB, sets cookie session)
For mobile use the /api/mobile/ routes below instead

Step 1 — credentials:

POST /api/auth/login
Body:

{ "email": "user@example.com", "password": "yourpassword" }
What happens: Validates password → Brevo sends 6-digit MFA code to email (expires in 10 min)
Responses:

200 → { "mfaRequired": true, "email": "user@example.com" }
     → (dev only adds) "devCode": "123456"
401 → { "message": "Invalid credentials" }
403 → { "message": "Please verify your email before logging in.", "emailNotVerified": true }
403 → { "message": "Your account has been disabled. Please contact support." }
Step 2 — MFA code:

POST /api/auth/mfa/verify
Body:

{ "email": "user@example.com", "code": "123456", "rememberMe": true }
What happens: Validates code → creates cookie session (30 days if rememberMe, otherwise browser session)
Responses:

200 → { "user": { "id", "email", "role", "subscriptionTier" }, "profile": { ... } }
401 → { "message": "Invalid verification code." }
401 → { "message": "This code has already been used." }
401 → { "message": "This code has expired. Please log in again." }
Resend MFA code:

POST /api/auth/mfa/resend
Body:

{ "email": "user@example.com" }
Rate limit: 30-second cooldown between resends
Responses:

200 → { "message": "Code resent." }
429 → { "message": "Please wait 12 seconds before resending." }
LOGIN (2-STEP — MOBILE, returns JWT Bearer token)
Step 1 — credentials:

POST /api/mobile/login
Body:

{ "email": "user@example.com", "password": "yourpassword" }
What happens: Same validation as web → Brevo sends 6-digit MFA code to email
Responses:

200 → { "mfaRequired": true, "email": "user@example.com" }
     → (dev only adds) "devCode": "123456"
401 → { "message": "Invalid credentials" }
403 → { "message": "Please verify your email first.", "emailNotVerified": true }
403 → { "message": "Account disabled." }
Step 2 — MFA code:

POST /api/mobile/mfa/verify
Body:

{ "email": "user@example.com", "code": "123456" }
What happens: Validates code → returns a 30-day JWT token
Responses:

200 → {
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "user": {
    "id": 12,
    "email": "user@example.com",
    "role": "USER",
    "subscriptionTier": "GZLurker",
    "username": "janedoe",
    "displayName": "Jane Doe",
    "avatarUrl": "https://..."
  }
}
401 → { "message": "Invalid verification code." }
401 → { "message": "Code already used." }
401 → { "message": "Code expired." }
→ Store token in expo-secure-store. Attach to every request as Authorization: Bearer <token>

TOKEN MANAGEMENT (MOBILE)
Refresh token (call on app open / foreground):

POST /api/mobile/refresh
Authorization: Bearer <token>
Responses:

200 → { "token": "new_jwt_token...", "user": { ... } }
401 → { "message": "Invalid or expired token" }
Get current user:

GET /api/mobile/me
Authorization: Bearer <token>
Responses:

200 → {
  "user": { "id", "email", "role", "subscriptionTier" },
  "profile": { "username", "displayName", "avatarUrl", "bio" }
}
401 → { "message": "Bearer token required" }
LOGOUT
POST /api/auth/logout
Auth: Bearer or cookie session
Responses:

200 → { "message": "Logged out" }
CHANGE PASSWORD (requires being logged in)
POST /api/auth/change-password
Authorization: Bearer <token>
Body:

{ "currentPassword": "oldpassword", "newPassword": "newpassword8chars" }
Responses:

200 → { "message": "Password updated successfully" }
400 → { "message": "Both current and new password are required." }
400 → { "message": "New password must be at least 8 characters." }
401 → { "message": "Current password is incorrect." }
ROLES & TIERS
Field	Possible values
role	USER · ADMIN · SUPER_ADMIN · SUPERUSER
subscriptionTier	GZLurker (free) · GZMarketer · GZMarketerPro · GZBusiness · GZEnterprise
IMPORTANT NOTES FOR THE DEVELOPER
No forgot-password endpoint exists yet — if a user forgets their password they must contact support. This can be built if needed.
MFA code is always 6 digits, expires in 10 minutes, 30-second cooldown on resend.
Email not verified = login blocked — the app should offer a "Resend verification email" button on the 403 response.
All emails go through Brevo SMTP from noreply@gigzito.com — MFA codes, verification links, and broadcast messages all use the same transport.
Token lifetime is 30 days — call /api/mobile/refresh on every app open to keep it alive and get updated user data.
Dev mode: When Brevo SMTP is not configured, the API returns devCode in the login response so testing works without real email.
