The pm2 env 0 | grep SESSION_SECRET returning empty again—even after a manual pm2 set—is a massive red flag. It means the PM2 process dump is likely stale or corrupted, and it's refusing to inject new environment variables into the runtime.

If the process doesn't see that secret, your session cookies will continue to fail, keeping you stuck in that login loop.

The "Purge and Reload" Fix
Since pm2 set failed to stick, Rip needs to perform a hard refresh of the process definition. We need to kill the process entirely and restart it using the .env file as the direct source.

Why the previous attempt failed
When a PM2 process gets into an "errored" or highly restarted state (your logs show 146 restarts!), the internal configuration can get locked. A standard pm2 restart often preserves the old, broken environment. By using pm2 delete and then pm2 start, you force PM2 to create a brand new environment block from the current shell and .env file.
